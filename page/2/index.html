java设计模式之外观模式

发表于 2018-02-12 | 更新于 2018-02-27 | 分类于 设计模式

定义

外观模式（Facade），为子系统中的一组接口提供一个一致的界面，定义一个高层接口，这个接口使得这一子系统更加容易使用。

适用

在遇到以下情况可以使用facade模式：

  1. 当你要为一个复杂子系统提供一个简单接口时。子系统往往因为不断演化而变得越来越复杂。大多数模式使用时都会产生更多更小的类。这使得子系统更具有可重用性，也更容易对子系统进行定制，但这也给那些不需要定制子系统的用户带来一些使用上的困难。facade可以提供一个简单的缺省视图。

  2. 客户程序与抽象类的实现部分之间存在着很大的依赖性。引入facade将这个子系统与客户以及其他的子系统分离，可以提高子系统的独立性和可移植性。

java设计模式之工厂模式

发表于 2018-02-12 | 更新于 2018-02-12 | 分类于 设计模式

说起工厂模式就不得不提起简单工厂模式，简单工厂模式又称静态工厂模式。简单工厂模式就是有一个工厂类根据传入的参数决定创建哪一种产品类。

简单工厂

简单工厂模式有四个角色

  • 工厂类角色：具体产品类角色直接调用者
  • 抽象产品角色：接口货抽象类，负责决堤产品角色的定义，及与客户端的交互
  • 具体产品角色：被工厂类创建的对象，也是客户端实际操作对象
  • 客户端：调用工厂类产生实例，并调用实例的方法进行相应的工作

Countdownlatch和Cyclicbarrier的区别

发表于 2018-02-11 | 更新于 2018-02-11 | 分类于 多线程

之前看过不少关于Countdownlatch和Cyclicbarrier的文章，也写过demo，总觉得自己理解了，一次偶然的机会在工作中用到了其中一个类，之后再回过头来理解这两个类会有不一样的理解，大多数人都是这样，只有在实际工作中用到了才会更加理解的透彻，因为可以结合具体的业务去使用。

业务背景

我当时的业务逻辑是这样的：用户在页面上上传一个Excel文档（Excel中的字段是做好给用户的），用户根据模板填写好内容进行上传，后台系统对用户填写的Excel数据进行校验；其中Excel中每行数据有25个字段，需要对每个字段进行校验，有的字段需要调用别的系统服务好几次，根据需要每个Excel用户最多可以上传200条数据，如果校验出的数据有误，需要将整个Excel中所有有误的数据一次性返回给用户，让用户一次性修改完成之后再上传，避免了用户多次上传的麻烦。

问题描述

以上是业务背景，起初业务量比较少，用户最多上传二三十条，每行数据校验的时间大概在50ms左右，这样起初如果只有二三十条数据的话其实总时间也还算还好，但随着业务量的增大，逐渐用户会上传200条数据，这时候就需要等待10s左右，这个时间就非常非常长了，第一次碰到瓶颈的时候是用户反馈页面页面提示一直在校验中（测试环境和预发环境是不存在这种情况的，毕竟这两个环境和生产环境还是有点区别的），经过一番排查最终定位到是生产环境中的Nginx设置的超时时间是5s，如果超过这个时间打开开发者调试工具会发现返回到页面的是timeout，但查看日志发现数据都是正常的。由此想到了用这两个类来优化校验时间。

关于哨兵的几个核心概念

发表于 2018-02-07 | 更新于 2018-02-07 | 分类于 高并发缓存

这篇主要介绍哨兵的几个概念：

sdown和odown转换机制

  • sdown是主观宕机，就一个哨兵如果自己觉得一个master宕机了，那么就是主观宕机

sdown达成的条件很简单，如果一个哨兵ping一个master，超过了is-master-down-after-milliseconds指定的毫秒数之后，就主观认为master宕机

  • odown是客观宕机，如果quorum数量的哨兵都觉得一个master宕机了，那么就是客观宕机

sdown到odown转换的条件很简单，如果一个哨兵在指定时间内，收到了quorum指定数量的其他哨兵也认为那个master是sdown了，那么就认为是odown了，客观认为master宕机

  • HashMap涉及的细节讨论

一、HashMap数据结构

HashMap的数据结构是由数组+链表组成

数组：采用一段连续的存储单元来存储数据。对于指定下标的查找，时间复杂度为O(1)；通过给定值查找，需要遍历数据的元素进行逐一对比，时间复杂度为O(n)

链表：对于链表的新增、删除操作（在找到指定位置后），仅需处理节点间的引用即可，时间复杂度为O(1)，而查找操作需要遍历链表进行逐一比对，时间复杂度为O(n)

所以为了快速定位并且能够快速的增删，使用了数组+链表的方式
