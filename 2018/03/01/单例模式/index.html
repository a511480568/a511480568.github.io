java设计模式之单例模式

发表于 2018-03-01 | 更新于 2018-03-01 | 分类于 设计模式

单例模式，是一种常用的软件设计模式。在它的核心结构中只包含一个被称为单例的特殊类。通过单例模式可以保证系统中，应用该模式的类一个类只有一个实例。即一个类只有一个对象实例。（百度百科）

单例模式主要有以下几种方式实现：

  • 饿汉式
  • 懒汉式
  • 内部类
  • 双重校验

饿汉式

饿汉式是在类加载时就进行了加载，有时我们希望在调用getInstance()时才会加载，这样才能达到lazy loading的效果，而懒汉式就满足了这一需求

懒汉式

懒汉式虽然能够在多线程中很好的工作，而且看起来也实现了lazy loading的效果，但我们不希望每次调用getInstance()时就进行同步加锁，这样会影响效率，为了解决效率低下的问题，可以采用静态内部类的方式来实现

内部类

静态同样利用了classloader的机制来保证初始化instance时只有一个线程，而且实现了当Singleton类被加载时并不会立即初始化instance，因为SingletonHandler并没有显示的调用，只有显示的调用getInstance()方法时才会装载SingletonHandler类并初始化instance实例。如果实例化instance很消耗资源，我想让他延迟加载，另外一方面，我不希望在Singleton类加载时就实例化，因为我不能确保Singleton类还可能在其他的地方被主动使用从而被加载，那么这个时候实例化instance显然是不合适的。

双重校验

单例模式的优缺点

优点

  • 由于单例模式在内存中只有一个对象，减少了内存开销，当一个对象需要频繁的创建、销毁时，单例模式能提升实例创建时的效率

  • 单例模式减少内存开销，当一个对象的产生需要比较多的资源时，可以考虑使用单例模式，创建一个单例对象常驻在内存中

  • 单例模式避免资源的多重占用

  • 单例模式可以在系统中设置全局访问点，优化和资源的共享访问。

缺点

  • 单例模式一般情况下没有接口，扩展困难

  • 单例模式与单一职责原则相冲突。一个类应该只实现一个逻辑，不需要关心它是否是单例模式。单例模式把单例和业务融合在了一起

应用

spring框架中的bean就是单例模式，这样做就是为了spring能够更好的对这些bean进行生命周期的管理，决定什么时候创建出来，什么时候销毁。
